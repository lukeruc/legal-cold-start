# 合同工作

## 工作前提

拿到一份合同后，在审阅任何一个条款之前，先与用户核实交易背景。以下信息决定后续所有判断的方向：

- **交易背景。** 什么交易、什么金额、什么时间表。
- **我方角色。** 发包人还是承包人、买方还是卖方、许可方还是被许可方。
- **对方属性。** 首次合作还是长期客户、国企还是民营、体量对比。
- **审查要求。** 常规审还是深度审、是否需要出具独立法律意见书、截止时间。

同一份合同条款，在不同背景下是不同结论。不核实背景就开始审，出来的意见可能是对的，但方向是偏的。

## 合同工作流程

### 一、判类型

先判定合同类型。施工合同和 NDA 的审查维度不同——判错了类型，审视的方向就歪了。

确定类型后，读取 `review/` 下对应的审查规则文件（如 `construction.md`、`nda.md`）。该文件定义了此类型合同应从哪几个维度审视。如果 `review/` 下没有对应文件，按通用法律逻辑审查，审查结束后建议将此类型的审查规则沉淀到 `review/` 下。

### 二、明情境

交易背景决定了同一套审查规则在应用时的严苛程度：

- **审核力度。** 大额首次合作不能只看表面合规，要深究条款背后的操作风险。小额日常合作重在快速拦风险，不在逐条博弈。
- **修改尺度。** 同一处不利条款，核心交易中要求修改，常规交易中可以标注风险后放过。
- **交付方式。** 常规审直接在文件上标注。深度审或对方难缠时，输出独立审核意见书。

情境不改变"查什么"，但改变"怎么查"和"怎么出结果"。

### 三、逐条审查

按该类型合同的审查维度逐条推进。每条给出：

- **条款现状。** 合同里写了什么。
- **风险评估。** 在当前交易背景下，这个条款可接受、需要改、还是有谈判空间。
- **修改建议。** 如果要改，建议改成什么措辞。

### 四、输出成果

根据情境决定交付方式。至少包含：

- **整体结论。** 这份合同在当前交易背景下，总体风险评级和能否推进。
- **逐条意见。** 每条关键条款的审查结果、风险等级、建议措辞。

深度审或重大交易时，增加一份独立法律意见书。

## 本目录结构

### review/ — 审查规则

按合同类型组织的审查维度文件。每个文件定义一种合同类型应从哪几个维度审视。

示例（`review/nda.md`）可能定义这些维度：保密信息定义范围、保密期限、使用限制、返还销毁义务、救济措施。示例（`review/construction.md`）则完全不同：签约主体资质、工程范围、工期约定、质量标准、合同价款与支付、违约责任、争议解决。

文件在使用中生长——审查几次同类型合同后，自然知道该加什么检查点。审查几次同类型合同后，自然知道该加什么检查点。

没有对应文件的合同类型，按通用法律逻辑审查。

### templates/ — 公司模板

存放公司自己起草合同用的模板文件。按合同类型组织，出站起草时直接使用。
